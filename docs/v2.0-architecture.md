# Solstice Season v2.0 Architecture Decisions

This document captures the architectural decisions for v2.0 of the Solstice Season integration.
Created: 2026-02-16

## Implementation Strategy

**Decision**: Start fresh from v1.5.1 tag, selectively copy good parts from v2-base-device branch.

**Rationale**: The v2-base-device branch introduced complex patterns (Singleton Base Coordinator,
Tracking-Sets, Device Registry Listener) that are difficult to clean up. Starting fresh and
copying only the valuable parts is cleaner.

### Code to Keep from v2-base-device

- `calculations.py` - Cross-Quarter and Chinese Solar Terms calculations (~lines 600-1131)
- Test files for Cross-Quarter and Chinese Solar Terms
- DataUpdateCoordinator pattern (but simplified, without singleton)
- UPDATE_INTERVAL = timedelta(days=1) scheduling

### Code to Remove / Not Copy

- Singleton Base Coordinator pattern (`async_get_or_create_base_coordinator()`)
- Hemisphere Select Entities (both `BaseHemisphereSelect` and `FourSeasonsHemisphereSelect`)
- Tracking-Sets (`_hemisphere_select_added`, `_base_sensors_added`)
- Device Registry Listener (`_setup_device_registry_listener()`)
- `async_remove_base_coordinator_if_unused()` cleanup logic

## Key Design Principles

### 1. Hemisphere is FIXED

**Decision**: Hemisphere is set once in ConfigFlow and cannot be changed at runtime.

**Rationale**:
- Runtime changes via Select Entity were confusing
- Users don't typically change hemisphere
- Simplifies code significantly (no need for refresh triggers, tracking)
- If user needs different hemisphere, they create a new device

### 2. No Automatic Device Creation

**Decision**: Base Data is an explicit, optional device type that users choose.

**Rationale**:
- Automatic singleton pattern was confusing ("why is this device here?")
- Users who don't need solar_longitude/daylight_trend don't get it
- Each device is explicitly requested by the user

### 3. Each Device is Independent

**Decision**: Each config entry creates one device with its own coordinator.

**Rationale**:
- No shared state between devices
- No complex cleanup logic when entries are removed
- Simple 1:1 relationship: config_entry → coordinator → device

## Sensors by Device Type

### Base Data
| Sensor | Description | Hemisphere-Dependent |
|--------|-------------|---------------------|
| `solar_longitude` | Ecliptic longitude of the Sun (0-360°) | No |
| `daylight_trend` | "increasing" or "decreasing" | Yes |
| `next_daylight_trend_change` | Next solstice timestamp | No |

### Four Seasons
| Sensor | Description |
|--------|-------------|
| `current_season` | spring/summer/autumn/winter |
| `next_season_change` | Next solstice/equinox timestamp |
| `days_until_next_season` | Days remaining |
| `season_progress` | Percentage through current season |

### Cross-Quarter
| Sensor | Description |
|--------|-------------|
| `current_period` | Imbolc/Ostara/Beltane/Litha/Lammas/Mabon/Samhain/Yule |
| `next_period_change` | Next cross-quarter event timestamp |
| `days_until_next_period` | Days remaining |

### Chinese Solar Terms
| Sensor | Description |
|--------|-------------|
| `current_term` | Current solar term (24 possible values) |
| `next_term_change` | Next solar term timestamp |
| `days_until_next_term` | Days remaining |

## Future Considerations

### Intelligent Update Scheduling

Current: `UPDATE_INTERVAL = timedelta(days=1)` - updates at arbitrary times.

Potential v2.0 enhancement: Update at midnight local time for clean day transitions.

```python
async def _async_update_data(self) -> SeasonData:
    # Calculate next midnight
    now = dt_util.now()
    next_midnight = now.replace(hour=0, minute=0, second=0) + timedelta(days=1)

    # Schedule next update for midnight
    self.update_interval = next_midnight - now

    return await self.hass.async_add_executor_job(...)
```

## Migration Path

For users upgrading from v1.x:

1. Existing "Four Seasons" entries continue to work (backward compatible)
2. Base Data sensors are NOT automatically created
3. Users who want solar_longitude/daylight_trend add "Base Data" device manually

## File Structure (Implemented)

```
custom_components/solstice_season/
├── __init__.py                  # Entry setup, routes by device type
├── config_flow.py               # Multi-step with Device Type selection
├── const.py                     # Constants including device types
├── calculations.py              # All astronomical calculations
├── manifest.json
│
├── # Coordinators (one per device type)
├── coordinator.py               # Four Seasons coordinator
├── base_data_coordinator.py     # Base Data coordinator
├── cross_quarter_coordinator.py # Cross-Quarter coordinator
├── chinese_coordinator.py       # Chinese Solar Terms coordinator
│
├── # Sensors (routed from sensor.py)
├── sensor.py                    # Entry point, routes by device type
├── base_sensor.py               # Base Data sensors
├── cross_quarter_sensor.py      # Cross-Quarter sensors
├── chinese_sensor.py            # Chinese Solar Terms sensors
│
└── translations/
    ├── en.json
    ├── de.json
    └── nl.json
```

**Key differences from v2-base-device**:
- No `select.py` - No runtime hemisphere changes
- No singleton pattern - Each config entry has its own coordinator
- No tracking sets or device registry listeners
